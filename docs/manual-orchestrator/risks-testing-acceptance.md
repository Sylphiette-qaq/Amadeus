# 风险、测试方案、验收标准与结论

## 14. 风险与应对

### 14.1 风险一：模型生成的 `function.arguments` 非法或不符合 Schema

应对：

- 强化工具描述与参数 Schema
- 对 `arguments` 做 JSON 解析和字段校验
- 增加参数错误重试
- 必要时采用更严格的 few-shot 示例

### 14.2 风险二：MCP 工具接入复杂度上升

应对：

- 第一阶段先打通本地工具
- 第二阶段再统一接入 MCP 工具
- 为每类工具建立标准适配层

### 14.3 风险三：多轮上下文快速膨胀

应对：

- 增加轮次上限
- 对工具结果做摘要化存储
- 引入历史压缩策略

### 14.4 风险四：流式与结构化解析冲突

应对：

- 将「决策轮」和「回答轮」拆分
- 决策轮不用流式
- 回答轮再流式输出

## 15. 测试方案

### 15.1 测试目标

测试方案的目标不是只验证「程序能跑通」，而是验证以下核心能力在去 Agent 化后仍然稳定成立：

1. 主编排程序能够正确驱动每一轮模型调用
2. 能够按 OpenAI 协议识别并处理 `assistant.tool_calls`
3. 工具执行结果能够通过 `role: "tool"` 正确回填
4. 普通问答、单工具调用、多轮工具调用和异常场景都可控
5. 发生异常时能够按预期终止、重试或降级

### 15.2 测试分层

建议将测试分为四层：

1. **单元测试**  
   验证编排器内部的纯逻辑模块，不依赖真实模型和真实外部工具。

2. **组件测试**  
   验证 ToolRegistry、ToolExecutor、MemoryStore、MessageBuilder 等模块之间的协作。

3. **集成测试**  
   验证主编排程序与模型接口、工具接口、上下文存储的完整链路。

4. **回归测试**  
   针对已有能力建立稳定样例，防止后续重构破坏基础对话和工具调用行为。

### 15.3 单元测试范围

建议优先覆盖以下函数或模块：

1. **工具调用判定**  
   输入 assistant 消息，验证能否正确识别是否存在 `tool_calls`。

2. **参数解析**  
   对 `tool_calls[].function.arguments` 做 JSON 解析，验证合法 JSON、非法 JSON、缺失字段、字段类型错误等场景。

3. **终止条件判断**  
   验证以下分支：
   - 有 `tool_calls` 时进入工具执行
   - 无 `tool_calls` 且 `content` 非空时结束
   - 无 `tool_calls` 且 `content` 为空时返回异常
   - 达到 `maxTurns` 时强制终止

4. **工具路由**  
   根据 `function.name` 能否正确找到本地工具或 MCP 工具。

5. **工具结果回填**  
   验证生成的 `role: "tool"` 消息是否包含正确的 `tool_call_id` 与 `content`。

6. **上下文拼装**  
   验证系统消息、历史消息、当前用户消息、工具结果消息的拼接顺序是否正确。

### 15.4 组件测试范围

建议增加以下组件级测试：

1. **`ToolRegistry`**  
   验证工具注册、重复名称冲突、禁用工具过滤、按名称查找是否正确。

2. **`ToolExecutor`**  
   验证正常执行、参数校验失败、工具不存在、执行超时、执行异常包装。

3. **`MemoryStore`**  
   验证消息写入、历史恢复、工具消息持久化、异常数据容错。

4. **`MessageBuilder`**  
   验证请求消息中是否正确包含：`system`、`user`、`assistant`、`tool`、`tools`。

### 15.5 集成测试场景

建议至少覆盖以下核心集成场景：

1. **普通问答不触发工具**  
   输入普通问题，模型返回无 `tool_calls` 的 assistant 消息，编排器直接输出最终答案。

2. **单次本地工具调用**  
   模型返回一个 `tool_call`，编排器执行本地工具并回填 `tool` 消息，下一轮获得最终回答。

3. **单次 MCP 工具调用**  
   模型返回一个 MCP 工具调用，编排器完成客户端调用并将结果回填。

4. **连续两轮工具调用**  
   第一轮 assistant 产生 `tool_calls`，工具回填后第二轮 assistant 再次产生 `tool_calls`，第三轮返回最终答案。

5. **多工具顺序调用**  
   同一轮 assistant 返回多个 `tool_calls`，编排器依次执行并回填，然后继续下一轮。

6. **上下文恢复后继续对话**  
   先执行一轮完整会话，持久化后重新启动程序，验证能否正确恢复上下文并继续对话。

### 15.6 异常与边界测试

这部分必须单独列为重点，因为去掉框架内置 Agent 后，异常处理责任完全转移到项目自身。

建议至少覆盖以下场景：

1. `function.arguments` 不是合法 JSON — 预期：拒绝执行工具，记录错误，并进入纠偏或失败分支。
2. `function.arguments` 缺少必填字段 — 预期：参数校验失败，不执行工具。
3. `function.name` 不存在 — 预期：返回工具不存在错误，并终止或要求模型改正。
4. 工具执行 panic 或返回 error — 预期：错误被统一包装，不导致主进程崩溃。
5. MCP 客户端初始化失败 — 预期：相关工具不可用，但主程序能够给出清晰错误。
6. MCP 工具调用超时 — 预期：按超时错误处理，并记录耗时与失败原因。
7. 模型连续返回空 `content` — 预期：触发异常响应保护，不进入死循环。
8. 超过最大轮次 — 预期：编排器强制终止，返回 `maxTurnsExceeded`。
9. 历史上下文文件损坏 — 预期：容错读取，必要时忽略非法记录，不影响程序启动。

### 15.7 Mock 策略

建议不要让绝大多数测试依赖真实 DeepSeek 接口，否则测试将不稳定且成本高。

推荐做法：

1. 为 ChatModel 抽象接口 — 测试中使用 fake model 或 mock model，按顺序返回预设 assistant 消息。
2. 为 ToolExecutor 抽象接口 — 测试中使用 mock executor，返回可控的工具结果和错误。
3. 为 MemoryStore 抽象接口 — 测试中优先使用内存实现，避免文件 IO 干扰大部分逻辑测试。
4. 仅保留少量真实集成测试 — 真实模型调用和真实 MCP 调用应作为手动测试或受控集成测试，不应成为默认单测路径。

### 15.8 测试用例组织建议

建议目录结构如下：

```text
internal/
├── orchestrator/
│   ├── orchestrator.go
│   ├── orchestrator_test.go
│   ├── parser_test.go
│   └── policy_test.go
├── tool/
│   ├── registry_test.go
│   ├── executor_test.go
│   └── mcp_test.go
└── memory/
    └── store_test.go
```

建议测试命名尽量体现业务意图，例如：

- `TestOrchestrator_ReturnsFinalAnswer_WhenNoToolCalls`
- `TestOrchestrator_ExecutesToolAndContinues_WhenAssistantReturnsToolCalls`
- `TestParser_ReturnsError_WhenArgumentsIsInvalidJSON`
- `TestExecutor_ReturnsTimeoutError_WhenToolExecutionExceedsDeadline`

### 15.9 验收前最小测试集

如果项目节奏较快，至少应在上线前完成以下最小测试集：

1. 普通问答无工具调用
2. 单次本地工具调用闭环
3. 单次 MCP 工具调用闭环
4. 连续两轮工具调用闭环
5. 非法 `arguments` 场景
6. 工具超时场景
7. 最大轮次终止场景
8. 历史上下文恢复场景

### 15.10 自动化建议

建议在后续 CI 中至少接入以下自动校验：

1. `go test ./...`
2. 编排器核心单元测试
3. 工具执行器组件测试
4. 一组不依赖外部网络的集成测试

对于依赖真实模型或真实 MCP 服务的测试，建议单独标记为手动测试或 nightly 测试，避免影响日常开发效率。

## 16. 验收标准

改造完成后，至少应满足以下验收条件：

1. 程序不再依赖 `adk.NewChatModelAgent`
2. 每次用户请求由项目自有主编排程序驱动
3. 能够正确识别 assistant 消息中的 `tool_calls` 与最终 `content`
4. 能够执行至少一个本地工具和一个 MCP 工具
5. 工具结果能作为下一轮上下文回填
6. 支持最大轮次限制与异常终止
7. 会话日志中可以看到每一轮模型输出和工具执行结果
8. API Key 不再硬编码在源码中

## 17. 结论

从当前项目实现看，去掉框架内置 Agent、改为「纯模型能力 + 项目自有主编排程序」的方向是合理的，而且改造成本可控。

本项目已经具备以下可复用基础：

- 已接入 ChatModel
- 已接入本地工具
- 已接入 MCP 客户端
- 已具备基础上下文存储

因此本次改造的核心不在于替换底层依赖，而在于收回执行控制权，并建立一层清晰、可调试、可治理、可扩展的编排层。

建议按「先最小替换主链路，再逐步增强工具注册、上下文结构化、可观测性」的方式分阶段推进，避免一次性重写导致风险失控。
