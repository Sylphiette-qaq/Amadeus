# 当前实现与问题

## 1. 文档目的

本文档用于说明 Amadeus 当前基于 CloudWeGo Eino 内置 Agent 的对话执行模式，及其改造为「仅使用框架提供模型能力，由业务侧主编排程序手动处理每一轮 LLM 输出」的技术方案，供评审与实施参考。

本次改造目标不是替换底层模型框架，也不是移除现有工具体系，而是将「推理轮次控制权、工具调用调度权、上下文拼装权、终止判断权」从框架内置 Agent 收回到项目自身代码中。

## 2. 当前实现现状

### 2.1 当前入口与执行链路

当前程序主入口位于 `main.go`，执行链路如下：

1. 启动时调用 `agent.GetAgent(ctx)` 创建 `adk.ChatModelAgent`
2. 主循环读取用户输入
3. 调用 `utils.StreamResponse(ctx, agent, userQuestion)` 执行一次对话
4. `StreamResponse` 从本地文件加载历史上下文并拼接当前用户消息
5. 调用 `ag.Run(ctx, agentInput)` 交给框架内置 Agent 处理
6. 框架内部决定是否调用工具、如何继续下一轮推理以及何时结束
7. 最终输出由 `utils.HandleMessageOutput` 负责打印并保存助手回复

### 2.2 当前核心文件

- `main.go`：程序入口，负责循环读取输入并驱动对话
- `agent/agent.go`：初始化 DeepSeek ChatModel、加载 MCP 工具、创建 `adk.ChatModelAgent`
- `utils/stream.go`：封装单轮对话执行与流式输出处理
- `utils/memory.go`：历史消息持久化与恢复
- `tools/mcpClient.go`：从配置创建 MCP 客户端
- `tools/calculator.go`：本地计算器工具

### 2.3 当前架构特点

当前项目中，框架承担了以下职责：

- 模型调用
- 工具绑定
- 工具调用决策
- 多轮推理编排
- 终止条件判断
- 事件流输出组织

项目自身只负责：

- 读取用户输入
- 拼接会话历史
- 展示输出
- 保存最终文本消息

## 3. 当前模式存在的问题

当前模式能够快速跑通工具型 Agent，但对后续演进存在以下限制：

### 3.1 对执行链路缺乏精细控制

一旦调用 `adk.ChatModelAgent.Run`，每轮模型输出、是否进入工具调用、工具结果如何回填、何时结束，都由框架内部控制。项目无法稳定插入以下能力：

- 自定义轮次上限
- 工具白名单/黑名单动态控制
- 不同轮次不同提示词策略
- 工具调用前置校验
- 工具调用后结果清洗与标准化
- 特定错误场景的重试、降级、熔断

### 3.2 可观测性不足

当前仅能消费 Agent 输出事件，难以清晰区分：

- 本轮模型原始输出
- 是否产生工具调用意图
- 工具参数提取结果
- 工具执行成功/失败原因
- 工具回填后的下一轮提示

这会直接影响调试效率和后续问题定位。

### 3.3 难以实现稳定的业务编排

如果后续需要引入以下能力，内置 Agent 模式会越来越受限：

- 人工审核节点
- 指定工具优先级
- 工具调用预算控制
- 多工具串行/并行策略
- 工具调用审计日志
- 对话状态机
- 业务规则驱动的系统提示动态拼装

### 3.4 上下文治理能力弱

当前历史消息通过文本文件全量恢复，再直接拼入模型输入，缺少：

- 上下文窗口裁剪
- 系统消息与业务消息分层
- 工具消息结构化保存
- 中间轮次摘要
- 历史压缩策略

### 3.5 安全与配置治理存在隐患

当前 `agent/agent.go` 中直接硬编码 API Key，不适合作为长期实现。改造时应一并调整为环境变量或配置文件注入。
