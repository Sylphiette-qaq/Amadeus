# Amadeus 手动编排改造执行计划

## 1. 文档定位

本文档基于 `docs/manual-orchestrator/` 下的技术方案整理而成，用于指导 Amadeus 从框架内置 Agent 模式迁移到「纯模型能力 + 项目自有主编排程序」模式的持续实施、跟踪和验收。

与技术方案文档不同，本文档重点回答以下问题：

- 先做什么，后做什么
- 每个阶段的完成定义是什么
- 当前进展如何持续维护
- 风险、依赖和验收如何落地

## 2. 目标与边界

### 2.1 改造目标

本次改造目标是将以下控制权从框架内置 Agent 收回到项目代码：

- 推理轮次控制
- 工具调用调度
- 上下文拼装
- 终止条件判断
- 结构化执行日志输出

### 2.2 不在本次范围内

以下事项不作为本轮改造的前置目标：

- 替换底层 ChatModel 或模型服务商
- 移除现有工具体系
- 一次性完成全部目录重构
- 一次性引入复杂状态机或并行工具调度

## 3. 执行原则

- 先替换主链路，再逐步增强工具、上下文和可观测性。
- 优先保证闭环可运行，再追求结构优化。
- 每一阶段必须具备明确入口、退出条件和可验证结果。
- 新旧能力切换期间避免大规模并发重构，控制变更面。
- 所有新增行为尽量可测试、可回滚、可观测。

## 4. 当前基线

截至当前技术方案定义，项目基线如下：

- 入口在 `main.go`
- 现有对话主链路依赖 `adk.ChatModelAgent`
- 已具备 DeepSeek ChatModel 接入能力
- 已具备本地工具与 MCP 客户端基础
- 已具备基础上下文持久化能力
- 存在 API Key 硬编码、上下文结构弱、编排不可控等问题

本执行计划默认以该基线为起点推进。

## 5. 里程碑总览

| 里程碑 | 目标 | 预期产出 | 完成判定 |
|------|------|------|------|
| M0 | 基线确认与方案落盘 | 执行计划、任务拆分、现状清单 | 范围、依赖、风险已确认 |
| M1 | 主链路去 Agent 化 | 可运行的手动编排闭环 | 不再依赖 `adk.NewChatModelAgent` 驱动请求 |
| M2 | 工具与上下文结构化 | 统一工具注册执行、结构化会话存储 | 本地工具和 MCP 工具都能接入统一链路 |
| M3 | 可观测性与策略治理 | trace、审计、限制策略、基础压缩 | 执行过程可排查、可限制、可回溯 |
| M4 | 验收与切换 | 测试通过、文档收口、旧链路下线 | 满足验收标准并可稳定交付 |

## 6. 分阶段执行计划

## 6.1 M0：基线确认与实施准备

### 目标

在改代码前完成实施准备，避免边改边补方案。

### 核心任务

- 确认当前主入口、Agent 初始化点、流式输出入口、记忆存储入口。
- 识别所有与 `adk.ChatModelAgent` 强耦合的代码位置。
- 明确第一阶段保留项：DeepSeek ChatModel、本地工具、MCP 客户端、现有 CLI 交互方式。
- 确认配置治理方案，至少包含 API Key、Base URL、模型名、最大轮次、工具超时。
- 建立本执行计划文档作为单一维护入口。

### 输出物

- 本执行计划文档
- 受影响模块清单
- 阶段任务清单

### M0 已确认基线

- 当前 CLI 主入口为 `main.go` 中的 `main()`。
- 当前对话执行链路为 `main.main -> agent.GetAgent -> utils.StreamResponse -> ag.Run -> utils.HandleMessageOutput`。
- 与 `adk.ChatModelAgent` 强耦合的核心位置为：
  - `agent/agent.go:GetAgent`
  - `utils/stream.go:StreamResponse`
  - `utils/output.go:HandleMessageOutput` 及其依赖的 ADK 输出结构
- 当前流式输出入口位于 `utils/stream.go` 中的 `ag.Run(ctx, agentInput)`，消费逻辑位于 `utils/output.go`。
- 当前上下文读写入口位于 `utils/memory.go`，底层文件为 `checkpoints/context.txt`。
- 当前本地工具与 MCP 工具都在 `agent.GetAgent` 中完成装配，不具备独立注册表与执行器。
- M1 可优先复用 `GetChatModel`、本地工具实现、MCP 客户端初始化逻辑、CLI 输入层和现有 memory 接口。
- `GetAgent`、`StreamResponse`、`HandleMessageOutput` 和纯文本上下文格式不适合作为长期实现，应分别在 M1/M2 重构。
- 当前存在明确配置与安全问题：
  - `agent/agent.go` 中存在硬编码 DeepSeek API Key
  - `tools/toolsConfig.json` 中存在硬编码 MCP Provider Key
  - `utils/memory.go` 会将对话内容明文落盘

### M0 结论

- M0 所需的影响面识别已经完成。
- M1 的最小改造范围已经明确，具备进入主链路替换的条件。
- 配置外置与密钥移除必须作为并行硬要求推进，不能后置到验收前再处理。

### 退出条件

- 已明确 Phase 1 的最小交付范围
- 已明确必须修改的模块和可以后移的模块
- 已确认配置外置是硬性要求

## 6.2 M1：主链路去 Agent 化

### 目标

替换现有 `ChatModelAgent.Run` 主链路，建立最小可运行的手动编排闭环。

### 核心任务

- 保留纯 ChatModel 初始化能力，拆出 `GetChatModel(ctx)`。
- 停止在请求主链路中创建和使用 `adk.ChatModelAgent`。
- 新增 `orchestrator` 主编排入口，负责单次请求的循环执行。
- 按 OpenAI `tools` / `tool_calls` 协议处理模型输出。
- 支持单工具串行调用。
- 将工具结果按 `role: "tool"` 回填模型上下文。
- 增加最大轮次限制与异常终止。

### 建议拆分

1. 模型初始化改造
2. 编排器主循环落地
3. 工具调用解析与回填
4. CLI 接入编排器
5. 最小闭环验证

### 当前目录落位

当前实现已按目标文档推荐方向完成第一轮目录迁移：

- CLI 入口迁移到 `cmd/amadeus/main.go`
- 模型初始化迁移到 `internal/model/chat_model.go`
- 编排层拆分到：
  - `internal/orchestrator/orchestrator.go`
  - `internal/orchestrator/loop.go`
  - `internal/orchestrator/parser.go`
  - `internal/orchestrator/policy.go`
- 工具体系迁移到：
  - `internal/tool/registry.go`
  - `internal/tool/executor.go`
  - `internal/tool/mcp.go`
  - `internal/tool/calculator.go`
- 记忆层迁移到：
  - `internal/memory/store.go`
  - `internal/memory/serializer.go`
- 会话状态已新增到 `internal/session/state.go`
- 终端交互已迁移到：
  - `internal/presentation/input.go`
  - `internal/presentation/output.go`

当前仍保留 `utils/` 下部分旧文件，仅作为未清理的兼容/遗留代码，不再承载新的主链路。

### 退出条件

- 用户请求由项目自有编排器驱动
- 无工具问答可以正常完成
- 单次本地工具调用可以形成闭环
- 超过最大轮次时可终止
- 代码中不再以 `adk.NewChatModelAgent` 作为主执行路径

## 6.3 M2：工具与上下文结构化

### 目标

将工具体系和会话存储从“可跑”提升到“可维护”。

### 核心任务

- 引入 `ToolRegistry` 统一管理工具元数据。
- 引入 `ToolExecutor` 统一负责参数校验、超时、错误包装和执行结果。
- 将 MCP 工具转换为统一注册结构。
- 将会话存储升级为 JSON Lines 或等价结构化格式。
- 支持结构化消息类型，至少覆盖 `system`、`user`、`assistant`、`tool`。
- 为工具消息补齐 `tool_call_id`、工具名、参数、结果、trace 信息。

### 退出条件

- 本地工具和 MCP 工具可通过统一执行入口调用
- 会话存储可以表达工具调用过程
- 历史恢复不再仅依赖纯文本拼接
- 参数错误、工具不存在、超时等错误具备统一返回结构

## 6.4 M3：可观测性与策略治理

### 目标

补齐线上可维护性，避免编排逻辑成为黑盒。

### 核心任务

- 为每轮模型调用增加 trace 日志。
- 记录工具调用审计信息，包括工具名、参数、耗时、结果和错误。
- 支持工具白名单/禁用开关。
- 增加重试、超时、连续异常保护等策略。
- 增加上下文窗口裁剪与工具结果摘要能力。
- 视复杂度选择是否引入策略模块，如 `policy.go`。

### 退出条件

- 每次请求都可追踪轮次级执行记录
- 工具调用失败原因可定位
- 上下文膨胀具备基础治理策略
- 核心限制项可通过配置调整

## 6.5 M4：验收、切换与收口

### 目标

完成测试、文档和旧链路清理，确保改造可以稳定落地。

### 核心任务

- 补齐最小验收测试集。
- 完成配置清理，移除源码硬编码密钥。
- 更新 README 或使用说明，说明新链路行为。
- 评估是否下线旧入口和兼容逻辑。
- 回收已废弃的 Agent 相关包装代码。

### 退出条件

- 满足本文第 10 节验收标准
- 关键自动化测试通过
- 旧链路是否保留已明确

## 7. 工作分解结构

### 7.1 代码改造清单

| 模块 | 主要动作 | 优先级 |
|------|------|------|

## 10. 验收标准

满足以下条件才可视为本次改造完成：

- 程序主链路不再依赖 `adk.NewChatModelAgent`
- 每次请求由项目自有主编排程序驱动
- 能正确处理 assistant 消息中的 `tool_calls`
- 至少支持一个本地工具和一个 MCP 工具接入统一链路
- 工具结果能够以 `role: "tool"` 回填下一轮上下文
- 支持最大轮次限制、异常终止和基础错误包装
- 会话或 trace 中可以看到轮次级模型输出与工具执行结果
- API Key 不再硬编码在源码中
- 最小测试集执行通过

## 11. 风险台账

| 风险 | 影响 | 应对措施 | 当前状态 |
|------|------|------|------|
| 源码中存在硬编码密钥 | 存在泄漏风险，阻碍配置治理 | 在 M1 并行完成环境变量化与配置外置 | 已识别 |
| 对话内容明文落盘 | 存在敏感信息持久化风险 | M1 保留最小实现，M2 升级结构化存储并补充治理策略 | 已识别 |
| `function.arguments` 非法 | 工具无法安全执行 | JSON 解析校验、参数纠偏、限制重试次数 | 持续关注 |
| MCP 接入复杂度偏高 | M2 进度受阻 | 先打通本地工具，再做统一适配 | 持续关注 |
| 上下文增长过快 | 成本升高、响应变差 | 轮次限制、摘要、窗口裁剪 | 持续关注 |
| 流式输出与结构化解析冲突 | 输出体验或稳定性下降 | 决策轮 `Generate`，最终轮 `Stream` | 持续关注 |
| 改造期新旧链路并存 | 维护成本上升 | 明确兼容期限，逐步收口 | 持续关注 |

## 12. 推荐实施顺序

1. 先完成 M0 的影响面确认和配置治理方案。
2. 立即推进 M1，优先拿到可运行的去 Agent 化闭环。
3. 在主链路稳定后进入 M2，统一工具和上下文模型。
4. 然后推进 M3，补足日志、策略和治理能力。
5. 最后以 M4 收口测试、文档和旧链路清理。
